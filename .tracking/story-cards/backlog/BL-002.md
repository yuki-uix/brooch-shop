# BL-002: 定制 AI 对话可回看（会话与消息留存）

## Summary

在「单次评估」之上，引入 **可持久化的多轮对话**（或至少 **用户与模型的消息时间线**），支持用户在定制流程中 **回看历史轮次**、上下文延续与客服排查；与 MVP 仅「一次提交 → 结构化结果」区分。

## PM

- **里程碑**: M4+（MVP 后迭代）
- **依赖建议**: T2.1–T2.2（定制请求与评估管线稳定）；可选与 `CustomRequest` 一对一或一对多会话关联
- **风险 / 备注**: 隐私与合规（图片/文字留存周期）、存储成本、PII 脱敏、与「报价快照锁定」规则的一致性（多轮后是否触发重新评估需产品规则）

## BA — 用户故事（草案）

作为 **用户**，我希望在定制询价过程中 **查看之前与 AI 的对话内容**，以便核对需求、对比不同说法下的结论，或在后续访问时 **接续上下文**。

### 验收标准（草案）

- [ ] 每条定制相关会话可关联唯一标识（如 `customRequestId` 或独立 `conversationId`）
- [ ] 消息列表包含：角色（用户/助手/系统）、时间、文本；若含图片引用则展示缩略图或链接
- [ ] 定制结果页或独立「对话记录」入口可 **按时间线回看**，支持基本滚动与加载（分页可选）
- [ ] 明确数据保留策略说明（前台文案或设置项草案）

## TL — 技术（草案）

- **数据模型（示意）**: `Conversation` / `Message`（或 JSON 列存于 `CustomRequest` 的轻量方案）；Prisma 迁移与查询索引（按 `customRequestId` + `createdAt`）。
- **API**: 如 `GET /api/custom-requests/:id/messages`、`POST .../messages`（多轮时）；与现有 `evaluate` 的调用关系需定义（评估是否仍为一轮 `generateObject` 或升级为 `streamText`/多步）。
- **不纳入（首版）**: 跨用户共享会话、实时协同、完整 RAG 知识库；与 BL-001（效果图）可并行规划但独立交付。

## UX（草案）

- 定制页结果态旁或二级页：**对话时间线**；移动端长列表可读性。
- 与 design-specs 后续增补线框对齐。

## Definition of done（进入开发时再细化）

- [ ] 拆分为具体 tech task / 测试用例（含留存与权限）
- [ ] dashboard backlog 与 epic 映射更新
