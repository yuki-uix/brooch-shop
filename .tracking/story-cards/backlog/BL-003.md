# BL-003: 真实支付接入（替换 Mock 结账）

## Summary

在现有「购物车 → 结账页 → 提交订单（Mock）」之上，接入 **真实第三方支付能力**（候选：微信支付 / 支付宝 / Stripe 等国际卡），完成 **预下单 → 调起支付 → 异步回调验签 → 订单支付状态落库**，替换当前无资金流的模拟成功路径。

## PM

- **里程碑**: MVP 后（建议独立发布，与 M4 可演示解耦）
- **依赖建议**: T3.1 / Epic 3 订单展示与混合行稳定；生产环境 HTTPS、可公网回调域名
- **风险 / 备注**: 合规与资质（境内支付需主体与产品类目）、手续费与对账、支付失败/超时/重复通知、退款与部分退款策略；定制行与标准品 **同一订单合并支付** 的幂等与金额校验

## BA — 用户故事（草案）

作为 **用户**，我希望在结账时使用 **真实支付方式** 完成付款，并在支付成功或失败后看到 **明确状态**，订单与金额与支付结果一致。

### 验收标准（草案）

- [ ] 结账页可选择支付方式（范围由选型决定：至少一种生产可用渠道）
- [ ] 调起官方收银台 / H5 / JS SDK（按平台能力），金额与订单号一致
- [ ] 支付成功：订单状态从「待支付」→「已支付」（或等价）；失败/取消可重试或重新下单
- [ ] 服务端 **仅依据支付渠道异步通知 + 验签** 更新支付状态，不信任纯前端成功回调
- [ ] 对账字段：第三方流水号、实付金额、支付时间可查询或导出（首版可简化为 DB 字段）

## TL — 技术（草案）

- **订单模型扩展**: `paymentStatus`（`pending` | `paid` | `failed` | `refunded` 等）、`paidAt`、第三方 `transactionId`；创建订单时可能先 `pending` 再支付，与当前「提交即 submitted」区分（需迁移策略）。
- **API**: 创建支付意图 / 统一下单接口；**Webhook / 异步通知** 路由（`POST /api/payments/.../notify`），幂等处理、签名校验。
- **安全**: 密钥仅存服务端环境变量；回调 IP 白名单（若渠道支持）；金额与 `orderId` 二次校验。
- **不纳入（首版）**: 分账、订阅、组合支付、跨境税务；完整财务对账系统。

## UX（草案）

- 结账页：支付方式选择、支付中 loading、失败原因与重试入口；订单详情展示支付状态与时间。
